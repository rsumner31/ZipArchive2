ptsochantaris commented Feb 17, 2015

Thanks for chiming in guys!

@brianmuse Much faster to swipe-open notification center and use the "x" button to dismiss all of them at once from Trailer's section header.

@mcos You can make notifications be temporary by changing them to "banners" from preferences -> notifications, although I think that then their auto-dismissal is too fast. There was no logic to actually expire notifications before, although after a certain amount of notifications OS X will expire them anyway.

@pedrocunha I think that used to be the default before, I haven't touched the setting, not sure why a lot of people are seeing this as a new behaviour - something Swift-related, perhaps, and before OS X must have been ignoring it. I've got a feeling this is not what most people want though, so I may change the default back to banners. Interesting thought on per-repo setting, will keep it in mind!

pedrocunha commented Feb 17, 2015

This is just food for thought but I think disabling comments completely (I know its an option and everyone is free to toggle it on/off) kinda defeats a bit what Trailer wants to achieve. Owner

ptsochantaris commented Feb 17, 2015

@pedrocunha I'm probably being a bit mentally hubristic here :) but I would say that what you are describing is exactly what @danielschwartz is trying to achieve in the first place by turning all comment notifications off - Viewing it from the perspective of your idea, in an environment where there are a ton of comment notifications, the Trailer menu becomes the "collapsed" pr notification list that you describe.

Gathering a lot of comments and issuing a notification is a thought, but then where would you draw the line? It becomes tempting for the user to set the "gathering" time threshold high, and then Trailer is useless to them, because other devs are shouting "didn't you see my comment?" in the chat by the time they have been notified anyway. In fact, the most busy the PR the less likely is that the user will find out that way.

Of course, this doesn't address the "one solitary comment" part of your idea which certainly has merit (i.e. "disable all comment notifications, but notify in case an "idle" PR gets a comment) - but would that be useful enough for you to warrant creating a feature request? If so, let's go for it! -- But for the purpose of this specific issue thread, I think the basic "mute all" option (or ideally, like you said, per-repo, which is a bit hard at this point) is the way to go.
Owner
